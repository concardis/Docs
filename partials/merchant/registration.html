Introduction

The Unified Settlement Report is a comma-separated values (CSV) file format that maps multiple settlement reports from external processors (for example, PayPal, Concardis, SEPA) into a unified format for all payment methods and processors.

This simplifies the reconciliation processes for merchants as they only have to handle one file format.

There is a separate settlement report file on a processor that may or may not be processed.

The files are processed on a daily basis.


General Formatting Instructions

You can distinguish between alphanumeric (AN) and numeric (N) parameters within the Unified Settlement Report.

PARAMETER TYPES DESCRIPTION SAMPLE
AN - Currency ISO 4217 - A3 currency code EUR
AN - rest Alphanumeric values, ​​coming from the Payengine, follow the format restrictions, defined by the Payengine API.
Alphanumeric values ​​from the processors are forwarded as they are.
N - Amounts Amount parameters use the "." as a decimal separator and can include as many decimal places as supported by the currency and provided by the processor.
There is no separate credit/debit sign, negative settlement amounts include a "-".		 -10.25
N - dates The date parameter follows the DDMMYYYY format. 15012019

NOTE

Even if there is a predefined Unified Settlement Report format, all these parameters will always be populated.

Specific instructions and mappings can be found in the corresponding subsections for the methods or processors.

In this section, you can see a list of all payment methods, the payment method type (credit card, SEPA, Sofort, Klarna, etc.), the status (active or inactive), the configuation and settings by payment method, and only in cases in which the payment method is a credit card - the configurations and settings by card brand, for the payment methods available in the Merchant Center.

NOTE Payment methods can be only activated by the Payengine admins.



PAYMENT INSTRUMENTS

Payment instruments store data specific to the payment method.

A payment instrument can be related to more than one customer and a customer can have more than one payment instruments.

An existing payment instrument can be used for recurring payments.


Create Payment Instrument

You can create a payment instrument in Merchant Center from the PAYMENT INSTRUMENTS tab.

To create a payment instrument, click the Create Payment Instrument button.

Result You will be redirected to a screen where you can select a payment method.

Select a payment method from the Payment method drop-down menu.

Provide the Merchant Payment Instrument ID information and click Save to validate the changes.


Manage Payment Instruments

You can use the drop-down Filters option to filter the results by date (Created at/Modified at), by Payment Instrument ID, by Merchant Payment Instrument ID and by Product.


CUSTOMERS


Create Customer

You can create a customer in Merchant Center.

To a customer account you can set addresses, personas, and payment instruments.

To create a customer, click the Create Customer button.

Result You will be redirected to a screen where you can provide your Email address, Merchant Customer ID, Customer Type, Addresses, and Personas.


Manage Customers


You can use the drop-down Filters option to filter the results by date (Created at/Updated at), by Email, by Merchant Customer ID, by Customer ID and by Address ID.

You can see the list of available customers displayed with the following parameters: ID, Email, Merchant Customer ID, Created at, and Modified at.

Click the Actions for more customer details, for example, ID, Email, Merchant Customer ID, Created at, Updated at, ADDRESSES, PERSONAS, and PAYMENT INSTRUMENTS

NOTIFICATIONS


Create Notification


Payengine supports predefined notifications via webhook (URL) and email for the most relevant events.

You can opt in to receive emails and manage your URL/Email notification preferences via Merchant Center.

To subscribe for notifications:

  1. Click Create Notification and fill in Notification Type *, Version, Name, Description and Events*
  2. Select all your notification preferences from the Events* drop-down menu.
  3. Check the Activate Notification checkbox to make the notification active, then Save to validate your changes.

Note The Activate Notification checkbox is checked by default.


Manage Notifications


You can use the drop-down Filters option to filter the results by date (Created at/Updated at), by Active or not, by Notification type (Email or URL, or Version - V1 or V2).


RISK SETTINGS


You can create a risk setting based on your preferences.

To add or edit a risk rule, select a rule type from the Rule Type* drop-down menu. (Available options are Complex Rule, Simple Rule, or Geolocation Rule.)

You can select a Simple Rule Rule Type, then select any payment method from the Payment Methods* section, and then you can specify the exact criteria to define a rule.

For example, for Giropay, you can specify that when the billing.address.country is different from the shipping.address.country you should receive a Red flag notification.

When you create a risk setting you can reject transactions based on your rules.


Create Risk Settings


Risk Cards


You can manage risk settings for a specific merchant only through Merchant Center and for global merchants through API.

There are three types of rules in Merchant Center => Risk Settings => Risk Card


Simple Rule


You can use the Simple Rule to compare two attributes.

For example if addressLastName is different than the BillingLastName, then the result should be in RED which means the transaction will fail.

You can select a Simple Rule Rule Type, then select any payment method from the Payment Methods * section, and then you can specify the exact criteria to define a rule.

For example, for Giropay, you can specify that when the billing.address.country is different from the shipping.address.country you should receive a Red flag notification.


Complex Rule


You can create a set of rules using the Complex Rule.

There is an option to choose which Payment Method you can restrict, for how long you can restrict it and the type of transactions you can include - AMOUNT or COUNT.

You can select a Complex Rule Rule Type, then select any payment method from the Payment Methods * section, and then you can specify the exact criteria to define a rule.

For example, if the amount of transactions per customer ID for Giropay is greater than 500 euro for more than 3 hours, a red notification will be fired.


Geolocation


You can use the Geolocation Rule to restrict your location.

You can select a Geolocation Rule Rule Type, then select any payment method from the Payment Methods* section, specify the Geolocations* and the Result* to define a rule.

For example, if you select Giropay as a Payment Method* and DE as a Geolocation, and a Red Result*, it means that when someone uses Giropay as a payment method and is from Germany (DE), a red notification will be fired.


Blacklists


You can manage a Backlist for the specific merchant in Merchant Center → Risk Settings → Blacklists.

You can create a blacklist appplicable to a specific merchant only and when the merchant tries to create a transaction with for example an address that is added in this table, then the transaction will fail.


The fields that can be restricted are:

  1. ClientName → first and last name of the customer (e.g Max Mustermann)
  2. Email
  3. IBAN
  4. BIN (orders) - first 6 numbers of the credit card
  5. BIN (payment instruments)
  6. IP
  7. CustomerId
  8. PaymentInstrumentId
  9. AddressId

Flexible 3DS


There is a Flexible 3ds tab in Merchant Center → Risk Settings → Flexible 3ds which applies to creditcard transactions only.

For example, 3DS will always be required for transactions in Germany greater than 1000 EUR.

The second rule means that we will never require 3DS for any country, for transactions less than 500 EUR.
